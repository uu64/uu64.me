先日 AWS Certified Solutions Architect Professional を受験し、無事合格した。

スコアは 790 / 1000 でボーダーが 750 だから結構ギリギリ。
Direct Connectやクラウド移行周りは実作業経験がなかったので苦労して、
なんとか勉強間に合ったという印象。

## 受験申し込み時の知識レベル

Webアプリケーション開発において業務でガッツリ使った経験があり、
試験範囲の50％程度は把握できていたような気がする。

- AWSでの業務仕様経験は約5年
- 主な用途はWebアプリケーションの開発で、ざっくりやっていたことは以下
  - EC2, ELB, RDS, S3 などを使ったモノリシックなWebアプリの構築
  - API Gateway, Lambda, CloudFront, ECS などを活用したサーバーレスアプリの開発
  - CloudFormation, Codeシリーズを使ったCICD基盤の構築
  - VPC周り（Subnet, Security Group...）やIAMなどのAWSアカウントの設計、構築

上記状態でAWSのサンプル問題を見て、正解率が6割前後、という状態だった。

## 勉強の進め方

筆者の場合、試験準備期間は約1ヶ月だった。
AWSは多種多様なサービスを提供しているので、それらすべてをサービスごとに深ぼっていては
試験に間に合いそうになかった。

試験要項を読むのはもちろんとして、まずはAWSが提供する教材で各サービスの特性について俯瞰し、
その後問題集をとにかく解いて足りない知識を補強する、という進め方をした。
使った教材について順に紹介する。

### AWS Exam Readiness

Exam ReadinessはAWSが提供する試験準備用の動画教材。日本語翻訳版があるのでそれを使った。
所要時間は半日程度。

https://aws.amazon.com/jp/about-aws/whats-new/2018/06/exam-readiness-courses/

最初に試験の構成について解説があり、
その後分野別（既存のソリューションの改善、新しいソリューションの設計、クラウド移行、コスト最適化、組織管理）の例題を解くという流れ。

例題数は各分野ごとに２〜３問なので多くはないのだが、
例題毎に AWS 公式コーチによりどのような判断軸で回答を絞ればよいのか、の解説があり、これが非常に役に立つ。

### AWS 公式資料

Exam Readinessの例題を解いていると、試験でよく出てくるサービスが何となく分かってくるので、
その中で自分の理解が浅いものや興味があるものについては [AWSサービス別資料](https://aws.amazon.com/jp/aws-jp-introduction/aws-jp-webinar-service-cut/) で
概要を把握する。

[AWS Well-Architected](https://aws.amazon.com/jp/architecture/well-architected/) の資料も目を通すとよい。

細かい部分はこの後問題集を使いながら知識を補強していけば良いので、
無理に全部読もうとせず「こんなサービスあるんだなあ」くらいの気持ちで流し読みするのがおすすめ。

### 公式の模擬試験

AWSが無料（！）で模擬試験を提供している。
試験直前に受けて得点率が低いと巻き返しが効かなくなるので、早めに受けて自分の現在地点を確認するのがおすすめ。

受験方法については以下の Classmethod さんの記事がわかりやすい。

https://dev.classmethod.jp/articles/new-aws-official-practice-questions/

### 問題集を解く

あとはひたすら問題集を解いて、足りない知識を補強する。
AWSは次から次へと新しいサービスが出てくるので、発行年が最近のものを選ぶと良いと思う。

筆者は以下を2周した。1周目は間違えた箇所の解説をよく読んで、2周目で定着しているか確認をした。

http://www2.ric.co.jp/create/book/contents/book_1248.html


## 感想

会社の補助金を申請して受けたので、受かってホッとした。
AWSのサービスを網羅的に把握する良い機会になったし、
今後サービスを設計する上での考え方の指針を得ることができた気がする。

何か新しいサービスの開発を始める時、実装レベルの細かい特徴が理解できていないことはよくあると思う。
そのような状況で最初の設計をしなければならない時、今回の試験で得たような視点でまずは判断することで、
性能や運用に関するリスクを最小化できるんだろう、と思った。
